## II. Teoría (documentar aquí)

Para **cada** estructura elegida, responde en este Markdown (Usa Enlaces permanentes a código para ilustrar tus respuestas):

II.a) ¿Qué tipo de herencia es? (Especialización, Extensión, Especificación, Construcción).  

RESPUESTA:
Primera propuesta:
Es una herencia por Especialización, las clases `Programador` y `Diseñador` son tipos más concretos de `Empleado`, ya que un programador es un empleado y un diseñador es un empleado. Además, ambas subclases redefinen el comportamiento del método `calcularSalario()`, adaptándolo a su lógica específica.

https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-Sromerop0610/blob/9c6405114224109493b3e6a8018ec71d293b7e40/src/main/kotlin/es/ies/ejercicios/u6/ej61/especializacion/Especializacion.kt#L3-L44

Segunda propuesta:
Es una herencia por Extensión, ya que las clases `CocheElectrico` y `Camion` amplían la funcionalidad de `Vehiculo` sin modificar su comportamiento base. No redefinen el método, sino que añaden nuevas funcionalidades propias.

https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-Sromerop0610/blob/9c6405114224109493b3e6a8018ec71d293b7e40/src/main/kotlin/es/ies/ejercicios/u6/ej61/extension/Extension.kt#L3-L27

II.b) ¿Quién es la superclase y cuáles son las subclases?

RESPUESTA:
Primera propuesta:
La superclase es `Empleado` y las subclases `Programador` y `Diseñador`. `Empleado` define el comportamiento común y las subclases especializan ese comportamiento

Segunda propuesta:

La superclase es `Vehiculo` y las subclases `CocheElectrico` y `Camion`. `Vehiculo` define el comportamiento común, y las subclases añaden funcionalidades adicionales sin modificar el comportamiento heredado.

II.c) ¿Existe otra forma de implementarlo que no sea herencia? (composición, interfaces, delegación, etc.)

RESPUESTA:
Primera propuesta:

Sí, podría implementarse usando la composición. Por ejemplo, en lugar de que `Programador` herede `Empleado`, podría tener un atributo de tipo `Empleado` dentro de la clase:
``` Kotlin
class Programador(
    val empleado: Empleado,
    val horasExtra: Int
)
```

Y también podría usarse una interfaz para definir el método `calcularSalario()` y que cada clase lo implemente de forma independiente.

Segunda propuesta:

Sí, podría implementarse mediante interfaces. Por ejemplo, `Vehiculo` podría ser una interfaz con el método `arrancar()`, y cada clase lo implementaría por separado.

También podría usarse la delegación, donde cada tipo de vehículo delega el comportamiento común en otra clase compartida.

II.d) ¿Qué ventajas aporta esta implementación con respecto a la alternativa?

RESPUESTA:
Primera propuesta:
La herencia por especialización permite:
- Reutilizar código común definido en la superclase.
- Aplicar polimorfismo fácilmente.
- Mantener una jerarquía clara y estructurada.

Frente a la composición, la herencia deja más clara la relación "es un" entre clases y facilita el uso de variables del tipo superclase.


Segunda propuesta:
La herencia por extensión permite:
- Compartir comportamiento común sin duplicar código.
- Añadir nuevas funcionalidades sin modificar la clase base.
- Mantener una estructura más organizada que si cada clase implementara todo desde cero.

Frente a usar solo interfaces, la herencia permite reutilizar implementación, no solo definir contratos.
