### S — SRP (Single Responsibility Principle)

**Código v0:** `es.ies.ejercicios.u6.ej65.srp`

Tarea:

- Identifica una clase/función que hace “demasiadas cosas”.
- Separa responsabilidades en tipos más pequeños (por ejemplo: normalización, formateo, persistencia, logging).
- Mantén un `main`/demo que pruebe el flujo.

Pista:

- Si una clase cambia por más de un motivo (cambiar el formato del informe, cambiar la regla de normalización, cambiar el logging, cambiar cómo se obtienen los datos), probablemente está violando SRP.
- En v0 fíjate en: preparación de datos + registro/normalización + generación de informe + salida por consola, todo junto. Intenta extraer al menos 2 componentes con nombres claros.

Preguntas:

1) ¿Qué responsabilidades tenía la versión v0?  

RESPUESTA:
La versión **v0** mezclaba varias responsabilidades dentro de una sola clase (`InformeAppServiceV0`).
Esta clase se encargaba de:

* **Preparar los datos** (crear las personas y alumnos).
* **Registrar personas** en `RegistroPersonas`.
* **Generar el informe** usando `InformeMarkdown`.
* **Mostrar mensajes por consola** (logs).
* **Buscar personas** en el registro.

Esto hace que una sola clase tenga demasiadas funciones diferentes, lo que **viola el principio SRP (Single Responsibility Principle)**.

2) ¿Qué clases/componentes has creado y qué responsabilidad tiene cada uno?

RESPUESTA:
Para aplicar el principio **SRP**, se han separado las responsabilidades en varias clases:

* **PreparadorDatos**
  Se encarga únicamente de **crear y proporcionar los datos iniciales** (personas y alumnos).

* **ServicioRegistroPersonas**
  Se encarga de **registrar personas y permitir buscarlas** dentro del `RegistroPersonas`.

* **GeneradorInforme**
  Se encarga exclusivamente de **generar el informe** utilizando `InformeMarkdown`.

* **Logger**
  Su responsabilidad es **mostrar mensajes por consola** (logs de la aplicación).

* **InformeAppService**
  Actúa como **coordinador** de los servicios anteriores. No realiza la lógica directamente, solo organiza el flujo del programa.

3) ¿Cómo mejora el mantenimiento/testeo con el refactor?

RESPUESTA:

El refactor mejora el mantenimiento y el testeo porque **cada clase tiene una única responsabilidad**.

Esto aporta varias ventajas:

* **Código más claro y organizado**, ya que cada clase hace solo una cosa.
* **Más fácil de modificar**, porque si cambia una funcionalidad (por ejemplo, la forma de generar el informe) solo hay que modificar la clase correspondiente.
* **Mejor para hacer pruebas**, ya que se pueden probar las clases de forma independiente (por ejemplo, probar solo el generador de informes sin depender del registro de personas).
* **Mayor reutilización**, porque los componentes pueden utilizarse en otras partes del programa.

En general, el código es **más mantenible, modular y fácil de ampliar**.


### O — OCP (Open/Closed Principle)

**Código v0:** `es.ies.ejercicios.u6.ej65.ocp`

Tarea:

- El generador v0 usa un `when` por “formato”.
- Refactoriza para que añadir un nuevo formato (p. ej. “HTML” o “JSON”) no requiera modificar el generador central.

Pista:

- Es normal que siga existiendo un punto donde “decides qué implementación crear” (CSV/Markdown/etc.).
- La idea de OCP es que esa decisión no obligue a modificar la **lógica principal** (el generador), sino que se haga fuera (por ejemplo en `main`, una factoría/registry o inyección de dependencias).

Preguntas:

1) ¿Qué parte violaba OCP y por qué?  

RESPUESTA:
La parte que violaba el principio OCP (Open/Closed Principle) era la clase GeneradorInformeV0, concretamente el when que dependía del enum.

Cada vez que se quería añadir un nuevo formato (por ejemplo HTML o JSON), era necesario modificar esta clase, lo que rompe el principio OCP, que dice que el código debe estar abierto a extensión pero cerrado a modificación.
https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-Sromerop0610/blob/576106221c83b7721a74be0b88662f09b2df82b1/src/main/kotlin/es/ies/ejercicios/u6/ej65/ocp/OcpV0.kt#L14-L19

El problema es que:

El when conoce todos los formatos.

Para añadir uno nuevo hay que:

modificar el enum

modificar el when

añadir el método correspondiente.

Esto hace que la clase no esté cerrada a modificaciones

2) ¿Qué técnica has usado (polimorfismo/estrategia/jerarquía) para cumplir OCP?

RESPUESTA:
Se ha utilizado polimorfismo mediante una jerarquía de clases, aplicando el patrón Strategy.

Primero se define una interfaz común para todos los formatos:
https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-Sromerop0610/blob/4fb5c1584aaabcf099c53ffb7283935b65b916cb/src/main/kotlin/es/ies/ejercicios/u6/ej65/ocp/OcpV1.kt#L9-L11

Después cada formato implementa esa interfaz:
https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-Sromerop0610/blob/4fb5c1584aaabcf099c53ffb7283935b65b916cb/src/main/kotlin/es/ies/ejercicios/u6/ej65/ocp/OcpV1.kt#L30-L38

El generador ahora trabaja con la abstracción, no con formatos concretos:
https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-Sromerop0610/blob/4fb5c1584aaabcf099c53ffb7283935b65b916cb/src/main/kotlin/es/ies/ejercicios/u6/ej65/ocp/OcpV1.kt#L44-L48

Gracias al polimorfismo, el generador puede usar cualquier formato que implemente FormatoInforme.

3) ¿Cómo añadirías un nuevo formato tras tu refactor?

RESPUESTA:

Para añadir un nuevo formato ya no es necesario modificar el generador, solo hay que crear una nueva clase que implemente la interfaz FormatoInforme.

Por ejemplo, para añadir un formato HTML:

```Kotlin
class FormatoHtml : FormatoInforme {
    override fun generar(titulo: String, items: List<Resumible>): String =
        buildString {
            appendLine("<h1>$titulo</h1>")
            appendLine("<ul>")
            for (item in items) {
                appendLine("<li>${item.resumen()}</li>")
            }
            appendLine("</ul>")
        }
}
```

Y usarlo en el programa:

```Kotlin
val generador = GeneradorInforme()
val formatoHtml = FormatoHtml()

println(generador.generar(formatoHtml, "Informe HTML", items))
```

De esta forma:

No se modifica GeneradorInforme

No se modifica código existente

Solo se añade una nueva clase

Esto cumple correctamente el principio OCP.

### L — LSP (Liskov Substitution Principle)

**Código v0:** `es.ies.ejercicios.u6.ej65.lsp`

Tarea:

- Hay una jerarquía donde una subclase rompe expectativas del tipo padre (comportamiento inesperado).
- Refactoriza para que cualquier subclase pueda sustituir al padre sin “sorpresas” (sin lanzar excepciones inesperadas, sin romper invariantes).

Pista:

- Si una “subclase” no puede cumplir el contrato del padre (por ejemplo, “solo lectura” no puede `guardar`), normalmente no debe heredar de ese tipo.
- Una solución habitual es separar interfaces por capacidades (lectura vs escritura) o usar composición para exponer solo lo que realmente se puede hacer.

### I — ISP (Interface Segregation Principle)

**Código v0:** `es.ies.ejercicios.u6.ej65.isp`

Tarea:

- La interfaz v0 es “gorda” (métodos no usados por algunos clientes).
- Divídela en interfaces pequeñas y adapta los clientes para depender solo de lo que usan.

Pista:

- Empieza por el cliente más pequeño: si un cliente solo necesita `buscar(...)`, no debería depender de `exportarCsv()` ni `borrarTodo()`.
- Segrega por capacidades (por ejemplo: lectura, escritura, exportación, borrado) y haz que cada cliente dependa solo de lo que usa.
- Señal típica de violación: ganas de dejar métodos vacíos, lanzar `UnsupportedOperationException` o meter `TODO(...)` al implementar.

Nota (relación con LSP):

- ISP y LSP suelen aparecer juntas: una interfaz “gorda” puede forzar a ciertos tipos a prometer operaciones que realmente no pueden cumplir, lo que termina rompiendo la sustitución (LSP).
- Segregar por capacidades (por ejemplo, lectura vs escritura) ayuda a que los tipos implementen solo contratos que pueden respetar, mejorando el cumplimiento de LSP.

Preguntas:

1) ¿Qué métodos eran irrelevantes para algún cliente?  

RESPUESTA:

El cliente BuscadorPersonasV0 solo necesitaba buscar personas, pero dependía de la interfaz RepositorioPersonasCompletoV0, que obligaba a tener métodos que no utilizaba.

Los métodos irrelevantes para ese cliente eran:

- guardar

- exportarCsv

- borrarTodo

3) ¿Qué mejora aporta a largo plazo?

RESPUESTA:
Aplicar ISP aporta varias mejoras importantes:

- Menor acoplamiento: los clientes dependen solo de los métodos que usan.

- Mayor flexibilidad: se pueden crear implementaciones parciales de las interfaces.

- Mejor mantenibilidad: cambiar una funcionalidad no afecta a clientes que no la usan.

- Mejor testeo: es más fácil crear mocks o implementaciones de prueba.

Por ejemplo, ahora podríamos crear un repositorio que solo permita buscar sin implementar otras operaciones:

```Kotlin
class RepositorioSoloLectura(
    private val datos: Map<String, Persona>
) : RepositorioBuscarPersonas {

    override fun buscar(nombre: String): Persona? = datos[nombre]
}
```

Esto sería difícil con la interfaz original, porque obligaría a implementar métodos innecesarios.
